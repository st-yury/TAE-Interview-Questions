1) A class should have only one reason to change.

2) It should have only one responsibility or job.

3) This principle promotes the idea that a class should be focused on doing one thing well.


Before SPR:
// Class violating SRP by handling both employee data and formatting
public class Employee
{
    public string Name { get; set; }
    public decimal Salary { get; set; }

    // Violating SRP by handling two responsibilities
    public void Save()
    {
        // Saving employee data to a database
        Console.WriteLine("Saving employee data to the database");
    }

    // Violating SRP by handling two responsibilities
    public void FormatAsJSON()
    {
        // Formatting employee data as JSON
        Console.WriteLine("Formatting employee data as JSON");
    }
}

class Program
{
    static void Main()
    {
        Employee employee = new Employee { Name = "John Doe", Salary = 50000 };

        // Saving and formatting are coupled in a single class
        employee.Save();
        employee.FormatAsJSON();
    }
}

After SPR:
// Class adhering to SRP by separating employee data and formatting responsibilities
public class Employee
{
    public string Name { get; set; }
    public decimal Salary { get; set; }
}

// Class responsible for saving employee data to the database
public class EmployeeRepository
{
    public void Save(Employee employee)
    {
        Console.WriteLine("Saving employee data to the database");
    }
}

// Class responsible for formatting employee data as JSON
public class JsonFormatter
{
    public string Format(Employee employee)
    {
        Console.WriteLine("Formatting employee data as JSON");
        // Actual JSON formatting logic would go here
        return $"{{\"Name\":\"{employee.Name}\",\"Salary\":{employee.Salary}}}";
    }
}

class Program
{
    static void Main()
    {
        Employee employee = new Employee { Name = "John Doe", Salary = 50000 };

        // Saving and formatting are now separated into different classes
        EmployeeRepository repository = new EmployeeRepository();
        repository.Save(employee);

        JsonFormatter formatter = new JsonFormatter();
        string json = formatter.Format(employee);
    }
}

1) Software entities (classes, modules, functions, etc.) should be open for extension but closed for modification.

2) This means that you should be able to add new functionality without altering existing code.

3) Achieved through the use of abstract classes, interfaces, and design patterns.

Before OCP:
// The Logger class directly writes to the console
public class ConsoleLogger {
  public void Log(string message) {
      Console.WriteLine(message); 
  }
}

// Usage
var logger = new ConsoleLogger();
logger.Log("Hello World!");

The ConsoleLogger class is not closed for modification. To add logging to a file, we would have to modify the class.

After OCP:

The Logger abstract class can be extended through inheritance.

public abstract class LoggerBase {
    public abstract void Log(string message);  
  }
  
  public class ConsoleLogger : LoggerBase {
      public override void Log(string message) {
          Console.WriteLine(message);
      } 
  }
  
  public class FileLogger : LoggerBase {
      public override void Log(string message) {
          // Write message to a file
      }
  } 
  
  // Usage
  LoggerBase logger = new ConsoleLogger(); 
  logger.Log("Hello World!");

The LoggerBase class is closed for modification but open for extension. We can create new loggers by inheriting from the LoggerBase without changing it. This follows OCP.

ISP splits interfaces to better model required functionality per type.

  void Eat();
  void Sleep();

  }

  }

  }

public class Robot : IWorkable

  {
  }

It suggests that high-level modules or classes should not depend on low-level modules but both should depend on abstractions. Additionally, abstractions should not depend on details; details should depend on abstractions. In simpler terms, DIP promotes the use of interfaces or abstract classes to decouple high-level and low-level components, allowing for flexibility and easier maintenance in software systems.

Before DIP:
class MailService
{
  public void SendMail(string message)
  {
      // Logic to send mail
  }
}

class UserService
{

  private MailService mailService = new MailService();

  public void NotifyUser(User user)
  {
      string msg = "Notification";
      mailService.SendMail(msg);
  }
}

After DIP:
interface IMailService
{
  void SendMail(string message);
}

class MailService : IMailService
{
  public void SendMail(string message)
  {
      // Logic to send mail
  }
}

class UserService
{
  private IMailService mailService;

  // Inject dependency 
  public UserService(IMailService mailService)
  {
      this.mailService = mailService;
  }

  public void NotifyUser(User user)
  {
      string msg = "Notification";
      mailService.SendMail(msg);
  }
}

The Don't Repeat Yourself principle aims to reduce duplication in code and improve maintainability and extendability.

Every piece of logical knowledge in a system should have one single representation.

Duplication of logic should be eliminated by abstracting it into reusable functions, classes, modules, etc.

Copying and pasting the same code structures or literals introduces unnecessary duplication.

Duplication leads to more places to change code when requirements change risking inconsistencies.

For example, hard-coding the same string literal like "User" in multiple places violates DRY. Instead it should be defined once as a constant "userStr" and reused.

The Keep It Simple, Stupid principle states that systems work best when they are kept simple rather than made complicated.

Simplicity should be a key goal in design and unnecessary complexity should be avoided.

Code should be kept clear, small and easy to understand.

Less code generally means less points of failure and fewer bugs.

Over-engineered solutions with too many extra features or moving parts should be avoided.

Code reuse and sharing logic avoids duplication and complexity.

Extra functionality, abstractions or architecture should only be added when justified by a real needed complexity.

GRASP is a collection of several key design principles in object-oriented development that deal with assigning responsibilities to classes and objects.

Applying these principles leads to flexible, reusable, and maintainable object-oriented code by properly assigning logical responsibilities among various classes and interfaces in the system.

"You Ain't Gonna Need It". It is a principle of software development that encourages developers to only add features to their code when they are actually needed by the users. This is in contrast to the traditional approach of overengineering code with features that might be needed in the future.
