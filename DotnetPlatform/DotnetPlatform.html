Multithreading is a technique for executing multiple tasks simultaneously within a single program. In .NET, multithreading is achieved by creating and managing multiple threads.

class Program
{
  public static void DownloadFile(string fileName)
  {
      // Simulate downloading a file
      Console.WriteLine("Downloading file: " + fileName);
      Thread.Sleep(3000); // Simulate download time
      Console.WriteLine("File downloaded: " + fileName);
  }

  public async static Task Main()
  {
      // Create two threads
      Thread thread1 = new(() => DownloadFile("file1.txt"));
      Thread thread2 = new(() => DownloadFile("file2.txt"));

      // Start the threads
      thread1.Start();
      thread2.Start();

      // Wait for both threads to finish
      thread1.Join();
      thread2.Join();

      Console.WriteLine("All files downloaded");
  }
}

/*
 * Downloading file: file1.txt
 * File downloaded: file1.txt
 * File downloaded: file2.txt
 * All files downloaded
 */

JIT compilation is a technique used by runtime environments, such as the Common Language Runtime (CLR) in .NET, to improve the execution performance of programs. Instead of compiling the entire program ahead of time into machine code, JIT compilation translates high-level code into machine code at runtime, just before it is executed.

C# code is compiled into an intermediate language called Common Intermediate Language (CIL or IL). This intermediate code is not platform-specific and is stored in assemblies.

When a .NET application starts, assemblies containing IL code are loaded into the runtime environment. The runtime doesn't immediately convert IL to machine code. Instead, it remains in IL form until needed.

When a method is called, the JIT compiler identifies the corresponding IL code. The first time a method is called, the JIT compiler translates its IL code into native machine code.

The generated native code is cached to avoid redundant compilation for subsequent calls to the same method. This cached native code is reused whenever the method is called again.

Value types:

  • Stored directly in memory:
    Value types store their data directly in the memory where the variable is allocated.
  • Contain the actual data:
    Variables of value types hold the actual data, and modifications to one variable do not affect others.
  • Ex.: int, float, char, struct.

int a = 10;
int b = a;
b = 20;

// 'a' remains 10 because 'b' is a copy of the value.

Reference types:

  • Stored in the heap:
    Reference types store a reference (memory address) to the location in the heap where the actual data is stored.
  • Pointers to data:
    Variables of reference types hold a pointer to the data, and modifications to one variable can affect others if they point to the same data.
  • Ex.: class, interface, object, string.

List<int> list1 = new List<int>() { 1, 2, 3 };
List<int> list2 = list1;

list2.Add(4);

// Both 'list1' and 'list2' reference the same list in memory.

Both list1 and list2 point to the same underlying data in the heap. Therefore, changes made through one variable are reflected in the other.

2. Enabling multiple iterations: IEnumerable allows obtaining multiple enumerators, so you can have multiple iterations over the same collection concurrently. If there was only IEnumerator, you could only iterate once before needing to reset.

using System.Collections;

namespace TestCodeRightHere;

internal class MyCollectionEnumerator : IEnumerator
{
    private List<int> internalList;

    int position = -1;

    public MyCollectionEnumerator(List<int> list)
    {
        internalList = list;
    }

    public bool MoveNext()
    {
        position++;

        return (position < internalList.Count);
    }

    public void Reset()
    {
        position = -1;
    }

    public object Current
    {
        get { return internalList[position]; }
    }
}

internal class MyCollection : IEnumerable
{
    private List<int> internalList = new();

    public IEnumerator GetEnumerator()
    {
        return new MyCollectionEnumerator(internalList);
    }

    public void Add(int item)
    {
        internalList.Add(item);
    }
}

class Program
{
    public static void Main()
    {
        // Usage example
        var collection = new MyCollection
        {
            1,
            2,
            3,
            4
        };

        // Enumerate multiple times concurrently
        var iter1 = collection.GetEnumerator();
        var iter2 = collection.GetEnumerator(); 

        iter1.MoveNext(); // 1
        iter2.MoveNext(); // 1

        Console.WriteLine(iter1.Current);
        Console.WriteLine(iter2.Current);

        iter1.MoveNext(); // 2
        iter2.MoveNext(); // 2

        Console.WriteLine(iter1.Current);
        Console.WriteLine(iter2.Current);

        iter1.MoveNext(); // 3
        // iter2.MoveNext(); still 2

        Console.WriteLine(iter1.Current);
        Console.WriteLine(iter2.Current);
    }
}

 * 1
 * 2
 * 2
 * 3
 * 2

namespace TestCodeRightHere;

internal interface IShape

internal class Circle : IShape

internal class Rectangle : IShape
{
    public int Area() => 100050;

internal class Triangle : IShape

internal interface IShapeCollection : IEnumerable
{
    void Add(IShape shape);

internal class ShapeCollection : IShapeCollection
{
    private readonly List<IShape> shapes = new();

    public IEnumerator GetEnumerator()
    {
        return new ShapeEnumerator(shapes);
    }

    public void Add(IShape shape)
    {
        shapes.Add(shape);
    }

internal class ShapeEnumerator : IEnumerator
{
    private List<IShape> shapes;

    public ShapeEnumerator(List<IShape> shapes)
    {
        this.shapes = shapes;
    }

    // Enumerator implementation

    int position = -1;

    public object Current
    {
        get { return shapes[position]; }
    }

    public bool MoveNext()
    {
        position++;

        return (position < shapes.Count);
    }

    public void Reset()
    {
        position = -1;
    }
}

4. Chaining/combining enumerables : With IEnumerable, you can chain/combine enumerable objects to iterate over all of them without needing to know the underlying enumerator type. This enables queries like LINQ. 

class Program
{
  public static void Main()
  {
      var numbers = new List<int>() { 1, 2, 3, 4, 10, 11, 12, 13, 301, 302, 303, 304 };

      var query = numbers
          .Where(n => n % 2 == 0)
          .Select(n => n * 2)
          .OrderBy(n => n)
          .Select((n, index) => new { Num = n, Position = index + 1 })
          .Concat(
              new List<int> { 10, 20, 30 }.Select(n => new { Num = n, Position = 0 })
          )
          .Where(x => x.Position > 0);

      foreach (var item in query)
          Console.WriteLine($"{item.Num} (Pos: {item.Position})");
  }
}

 * 8 (Pos: 2)

The Large Object Heap and Small Object Heap are two separate heaps that manage the allocation and garbage collection of objects in memory. They are designed to handle objects of different sizes and allocation patterns efficiently.

The SOH is divided into multiple generations, each with a different garbage collection policy. This helps to reduce the frequency of garbage collection and improve overall performance.

It is a separate heap from the SOH because large objects can cause fragmentation in the SOH and make garbage collection less efficient. The LOH is not compacted during garbage collection, which can lead to memory fragmentation over time. However, this also means that the LOH can handle larger objects without affecting the performance of the SOH.

SOH LOH
Object size < 85 KB > 85 KB
Garbage collection Compacted Not compacted
Performance More efficient Less efficient
Use cases Smaller objects, such as primitive data structures and method calls Large objects, such as images, files, and network buffers

GC generations are a concept employed by the .NET garbage collector to optimize the performance of object allocation and deallocation. The heap, where objects reside in memory, is divided into three generations: Generation 0 (G0), Generation 1 (G1), and Generation 2 (G2).

  • G0 is the youngest generation, typically containing short-lived objects that are created and destroyed frequently. Since these objects are expected to have short lifespans, G0 is garbage collected relatively often, minimizing the time spent searching for and reclaiming unused objects.
  • G1 is a buffer between G0 and G2. Objects that survive garbage collection in G0 are promoted to G1, indicating that they have exhibited some longevity. G1 receives less frequent garbage collections than G0, allowing the GC to focus on longer-lived objects.
  • G2 is the oldest generation, containing the most persistent objects. These objects are garbage collected less frequently than objects in G1, ensuring that the GC doesn't waste time scanning through objects that are unlikely to be freed.

Inheritance is a mechanism in object-oriented programming (OOP) that enables a class to derive properties and behavior from another class. The class that is being derived from is called the base class, and the class that is deriving from the base class is called the derived class. In C#, inheritance is implemented using the ":" symbol followed by the name of the base class.

To use inheritance effectively in C#, you should follow these best practices:

  • Use inheritance to model an "is-a" relationship: Inheritance should be used when the derived class "is a" type of the base class. For example, a Car "is a" Vehicle, so you can use inheritance to derive a Car class from a Vehicle class.
  • Avoid deep inheritance hierarchies: Deep inheritance hierarchies can make your code difficult to understand and maintain. You should try to keep the inheritance hierarchy as shallow as possible, ideally with no more than three levels of inheritance.
  • Favor composition over inheritance: In some cases, composition (using objects as member variables of a class) can be a better alternative to inheritance. This can help reduce the complexity of your code and make it easier to understand.
  • You should consider using composition over inheritance in C# in the following situations:
    • 1. When you want to reuse behavior across multiple classes, but there is no "is-a" relationship: Inheritance is appropriate when you have a base class that defines behavior that is common to all derived classes. However, if you have behavior that is shared between classes that do not have an "is-a" relationship, composition may be a better choice. For example, if you have a ShoppingCart class and a Product class, you could use composition to give the ShoppingCart a list of Products, rather than creating a derived class for each type of shopping cart.
    • 2. When you want to change behavior at runtime: With inheritance, the behavior of a derived class is determined at compile-time. If you need to change behavior at runtime, composition may be a better choice. For example, you could use composition to give a class a strategy object that can be changed at runtime to alter its behavior.

Severity and priority are two different concepts in software testing:

Severity refers to the degree of impact a defect or issue has on the system, product, or application being tested. Severity is usually measured on a scale from 1 to 5, with 1 being the most severe and 5 being the least severe. A severe defect could cause system failure or data loss, while a less severe defect might just cause a cosmetic issue or inconvenience for the user.

Priority refers to the order in which defects or issues should be addressed based on their impact on the project and the stakeholders involved. Priority is usually measured on a scale from 1 to 5, with 1 being the highest priority and 5 being the lowest priority. A high priority defect might need to be fixed immediately to avoid a major impact on the project, while a low priority defect could be addressed later when resources and time permit.

In general, severity indicates the impact of a defect on the product, while priority indicates the importance of fixing the defect relative to other tasks and priorities in the project.

Behavior-driven development (BDD) is a software development methodology that focuses on defining the expected behavior of a system in natural language, which can be understood by both technical and non-technical stakeholders. BDD involves collaboration between developers, QA testers, and business stakeholders to create a shared understanding of the requirements and desired outcomes of a system. BDD typically involves the use of a domain-specific language (DSL) that allows stakeholders to describe the desired behavior of the system in a structured format. The DSL is then used to generate automated tests that verify the system's behavior. BDD is often used in Agile development methodologies and is intended to promote collaboration and a shared understanding of the system's behavior among all stakeholders.

Sanity testing and retesting are two different types of testing that serve different purposes.

Sanity testing is performed after a new build is received to ensure that the most critical functionalities of the system are working as expected. It is a quick and focused test that checks whether the build is stable enough for further testing. Sanity testing is usually performed manually and does not cover all functionalities of the system.

Retesting, on the other hand, is performed to verify that a previously failed test case is working correctly after the defect that caused the failure has been fixed. Retesting is done to ensure that the defect has been resolved and the affected functionality is now working as expected. It is usually performed after the fix has been made and the system is rebuilt. Retesting can be performed manually or using automated test scripts.

In summary, sanity testing is done to check the stability of the new build, while retesting is done to verify that a previously failed test case is working correctly after the defect has been fixed.

Load testing and stress testing are both performance testing techniques used to evaluate the behavior of a system under specific workloads. However, they differ in the way they apply load to the system and the purpose of testing.

Load testing involves testing a system under a specific load, usually the expected or maximum user load, to determine its behavior, performance, and responsiveness. It measures the system's ability to handle the expected load and ensures that it is functioning as expected without crashing or slowing down.

Stress testing, on the other hand, is a technique used to test a system's ability to handle extreme workloads, beyond its design limits, and determine its breaking point. Stress testing is performed to identify the system's failure points and measure how it behaves under extreme conditions.

In summary, load testing checks the behavior of a system under expected conditions, while stress testing evaluates the system's performance under extreme conditions.

Usability testing and user acceptance testing are two different types of testing, with different objectives and methods.

Usability testing is performed to evaluate the ease of use and overall user experience of a product, usually a software application or a website. This type of testing involves observing and collecting feedback from users as they perform specific tasks on the product. The aim is to identify any issues with the product's usability, such as confusing navigation or difficult-to-use features, and to make recommendations for improvement.

On the other hand, user acceptance testing (UAT) is performed to determine whether a product meets the business requirements and is fit for use from an end-user perspective. UAT is typically carried out by the end-users themselves, who perform a series of predefined tasks to ensure that the product meets their needs and expectations. The focus is on validating the functionality and verifying that the product meets the business goals.

In summary, while usability testing is focused on evaluating the user experience of a product, UAT is focused on verifying that the product meets the business requirements and is fit for use from an end-user perspective.

Equivalence partitioning is a testing technique that involves dividing the input domain of a system into groups or partitions that are expected to behave similarly. The primary goal is to reduce redundant testing by selecting representative test cases from each partition. This technique is based on the assumption that if one test case within a partition exposes a defect, the other cases in the same partition are likely to do the same.

IEnumerable: When you perform LINQ operations on an IEnumerable, such as a List or an array, the entire dataset is loaded into memory first.
