SOCKS (Socket Secure, SOCKS4 and SOCKS5) is a networking protocol. It is like a secret tunnel that you can use to go online.

  1. The client sends a request to the SOCKS proxy.
  2. The SOCKS proxy authenticates the client.
  3. The SOCKS proxy establishes a connection to the server.
  4. The SOCKS proxy tunnels all traffic between the client and the server.
  5. The SOCKS proxy terminates the connection to the server.

It is a popular choice for tunneling traffic over a VPN, for bypassing firewalls, and for hiding the user's IP address.

  1. HTTPS is more widely supported than SOCKS. Most websites and applications support HTTPS, while only a few support SOCKS.
  2. HTTPS is simpler to set up and use than SOCKS. Most websites and applications automatically use HTTPS, while SOCKS requires manual configuration.
  3. If you use a VPN with SOCKS and still access a website that uses HTTPS, your traffic will be encrypted twice. The first layer of encryption will be provided by the VPN, and the second layer of encryption will be provided by HTTPS.
  4. If you are only using SOCKS to access a regular website that uses HTTPS, your traffic will be encrypted by the HTTPS protocol. However, your IP address will still be visible to the website. This means that the website will be able to see where you are connecting from.

The OSI (Open Systems Interconnection) model is a conceptual framework that standardizes the functions of a telecommunication or computing system into seven abstraction layers. These layers, from the physical transmission of bits to the user interface, help in understanding, designing, and troubleshooting network communication.

  • 1. Physical Layer (Ethernet cables, fiber-optic cables, and connectors): Deals with the physical connection between devices, including cables, connectors, and hardware.
  • 2. Data Link Layer (Network Interface Cards (NICs), MAC addresses, switches, and bridges): Responsible for creating a reliable link between two directly connected nodes, addressing, and error detection.
  • 3. Network Layer (Routers, IP addresses, and routing protocols (e.g., OSPF, BGP)): Manages the routing and forwarding of data packets between different networks, handling logical addressing.
  • 4. Transport Layer (Transmission Control Protocol (TCP) and User Datagram Protocol (UDP)): Ensures end-to-end communication by managing flow control, error correction, and data segmentation.
  • 4. Session Layer (NetBIOS (Network Basic Input/Output System)): Establishes, maintains, and terminates communication sessions, coordinating data exchange between applications.
  • 4. Presentation Layer (Encryption protocols (e.g., SSL/TLS), ASCII, JPEG, GIF): Handles data format translation, ensuring compatibility between different systems by dealing with encryption, compression, and encoding.
  • 4. Application Layer (HTTP, FTP, SMTP, and other application-level protocols): Provides network services directly to end-users, supporting application-level functionalities and user interfaces.

TCP:

  • 1. Connection-Oriented: Establishes a connection before data exchange.
  • 2. Reliable and Ordered: Ensures data integrity, sequencing, and error recovery.
  • 3. Flow Control: Adjusts data transmission rate based on receiver's capacity.
  • 4. Acknowledgments: Requires acknowledgment of received data.
  • 5. Usage Scenario: Best suited for applications where data integrity and completeness are critical, such as file transfer, email, and web browsing.

UDP:

  • 1. Connectionless: No connection setup before data transfer.
  • 2. Unreliable and Unordered: Does not guarantee data delivery or order.
  • 3. No Flow Control: Data sent without adjusting to receiver's capacity.
  • 4. No Acknowledgments: Does not require acknowledgment of received data.
  • 5. Usage Scenario: Suitable for real-time applications, streaming, online gaming, and scenarios where low latency is more important than guaranteed delivery.

An IP (Internet Protocol) address is a numerical label assigned to each device participating in a computer network that uses the Internet Protocol for communication.

It serves two principal functions: host or network interface identification and location addressing.

IPv4:

  • 32 bits.
  • Consists of four sets of numbers separated by dots (e.g., 192.168.0.1).
  • Limited due to the 32-bit address space, leading to IPv4 address exhaustion.

IPv6:

  • 128 bits.
  • Consists of eight groups of hexadecimal digits separated by colons (e.g., 2001:0db8:85a3:0000:0000:8a2e:0370:7334).
  • Vastly increased address space, designed to overcome IPv4 limitations.

The Domain Name System (DNS) translates human-readable domain names (like www.example.com) into IP addresses that machines use to locate each other on a network.

How it works:

  • 1. Domain Resolution Request: A user enters a domain name in a web browser.
  • 2. DNS Query: The system queries a DNS server to resolve the domain to an IP address.
  • 3. Recursive DNS Server: If the local DNS server doesn't have the information, it queries higher-level DNS servers recursively until it finds the authoritative DNS server for the domain.
  • 4. Authoritative DNS Server: Provides the IP address associated with the requested domain.
  • 5. Caching: The resolved IP address is often cached at different levels, speeding up future requests.
  • 6. Communication: The user's device can now communicate directly with the IP address associated with the entered domain, establishing the desired connection.

DHCP (Dynamic Host Configuration Protocol):

DHCP automatically assigns IP addresses and network configuration information to devices within a network, facilitating efficient and automated network setup.

When a device joins a network, it sends a DHCP request. A DHCP server responds with an IP address and other configuration details, ensuring seamless connectivity without manual configuration. A device broadcasts a DHCP discover message to identify available DHCP servers on the network. DHCP servers respond with DHCP offer messages, providing IP addresses and configuration details. The device selects an offer and sends a DHCP request to the chosen server. The chosen DHCP server acknowledges the request, assigning the device the offered IP address and relevant network settings.

Subnetting is the practice of dividing a larger network into smaller, more manageable sub-networks or subnets.

Enhances network efficiency, security, and management by logically grouping devices with similar IP addresses, reducing broadcast domains, and providing better control over network traffic.

NAT (Network Address Translation) is a technique that allows multiple devices on a local network to share a single public IP address for internet connectivity.

NAT enhances address space utilization, provides a layer of security by hiding internal IP addresses, and enables multiple devices within a private network to access the internet using a single public IP address.

A router is a networking device that connects different networks and directs data between them based on IP addresses. Routes data between networks, determines the best path for data transmission, and provides network layer (Layer 3) functionality.

A switch is a networking device that connects devices within the same local network, directing data based on MAC addresses. Switches operate at the data link layer (Layer 2) and facilitate communication between devices on the same network, improving local network efficiency.

A firewall create a barrier between internal private networks and external public networks like the internet. It filters and controls incoming and outgoing network traffic using a defined set of security rules. This allows protecting internal resources from unauthorized access as well unfiltered internet connections. Firewalls block cyber attacks, intrusion attempts, service exploits while allowing legit encrypted connections through VPN tunnels and secure protocols. By providing a checkpoint, firewalls are a critical building block in securing private enterprise and home networks.

Firewalls maintain logs of network activity, enabling administrators to monitor traffic, detect suspicious patterns, and investigate security incidents.

Firewalls regulate and control the incoming and outgoing network traffic based on predetermined security rules.

ARP (Address Resolution Protocol) is a protocol used to map a known IP address to a MAC (Media Access Control) address in a local network.

When a device wants to communicate with another device on the same network, it sends an ARP request to discover the MAC address associated with a known IP address. The device with the matching IP address responds with its MAC address, allowing the sender to create an ARP table entry for future communication.

Virtual Local Area Networks are a method of logically segmenting a physical network into multiple virtual networks, allowing devices to be grouped together as if they are on the same network, regardless of their physical location.

They are created by configuring network switches rather than physical rearrangements.

VLANs help control traffic, enhance security, reduce congestion, and simplify network management by compartmentalizing devices into logical groups even if they are not physically close within the network. Different VLANs cannot directly communicate without special routing configurations. This provides traffic isolation and containment that allows large networks to be partitioned efficiently. In summary, VLANs enable logically defined groups that have the same properties as traditional physical LANs within the same basic physical infrastructure.

Symmetric Encryption:

  • Single Key: Symmetric encryption uses a single shared key for both encryption and decryption. The same key is used by both the sender and the recipient.
  • Speed: It is generally faster than asymmetric encryption because of its simplicity and direct use of a single key.
  • Challenges: The primary challenge is key distribution — securely sharing the key between communicating parties can be vulnerable to interception.

Asymmetric Encryption:

  • Key Pairs: Asymmetric encryption uses a pair of keys: a public key and a private key. The public key is shared openly, while the private key is kept secret.
  • Encryption and Decryption: Data encrypted with the public key can only be decrypted with the corresponding private key and vice versa. This ensures that even if the public key is known, the private key remains secret.
  • Key Distribution: Asymmetric encryption helps address the key distribution challenge seen in symmetric encryption, as the private key never needs to be shared.
  • Complexity: Asymmetric encryption is computationally more intensive than symmetric encryption.

QoS (Quality of Service) refers to techniques for guaranteeing performance in networks by prioritizing certain types of traffic. It manages congestion through classifiers that categorize traffic and apply policies giving precedence to time-sensitive or business-critical applications. Methods like traffic shaping, prioritization, and resource reservation allow mission critical network traffic to take priority, especially during bandwidth congestion. QoS aims to provide consistent and reliable data delivery for key services by dedicating bandwidth relative to an organization's needs.

QoS improves performance by setting rules for which applications get priority access to network bandwidth to meet performance needs.

A network topology refers to the physical and logical layout of a network - it defines how devices connect and communicate with each other.

The topology affects transmission speed, latency, throughput and determines reliability if links fail. Factors like cost, capacity, extendability and ease of troubleshooting determine which works best. Understanding topologies allows optimizing network performance and efficiency.

Some common topologies are:

  • Bus: All devices connect to a common backbone in a single line.
  • Star: Devices connect to a central hub or switch.
  • Ring: Nodes connect to each other in a circular loop.
  • Mesh: Nodes connect directly to as many other nodes as possible.

The star topology is the most popular and commonly used network topology among regular internet users.

In a star topology, all devices and nodes in the network connect to a central device like a router, hub, or switch that functions as a central node facilitating communication. This central device acts as an intermediate for data transmission between the networked devices.

Simple to install and wire. No disruptions when adding/removing devices.

Centralized control is easy to manage and troubleshoot issues.

Fault isolation - other nodes not affected if one node fails.

Works well in small to mid-size networks.

Broadband router is an affordable central device option.

Transport Layer Security (TLS) is a cryptographic protocol that provides communications security over a computer network. It is the successor to the older Secure Sockets Layer (SSL) protocol and is widely used to protect sensitive data, such as login credentials, credit card numbers, and personal information, from being intercepted by unauthorized parties.

  • 1. Handshake Initiation: The TLS handshake begins when a client (e.g., a web browser) initiates a connection to a server that supports TLS. The client indicates its intent to establish a secure connection.
  • 2. Server Response: The server responds by presenting its digital certificate to the client. The certificate includes the server's public key and is signed by a trusted certificate authority (CA). The client checks the certificate's validity.
  • 3. Key Exchange: The client generates a pre-master secret and encrypts it with the server's public key from the presented certificate. This encrypted pre-master secret is sent back to the server.
  • 4. Decryption and Key Derivation: The server decrypts the pre-master secret using its private key. Both the client and server independently generate a session key from the pre-master secret. This session key will be used for symmetric encryption during the session.
  • 5. Authentication and Key Exchange: The client and server exchange a series of messages to authenticate each other and agree on the algorithms and parameters for the secure session. This includes verifying the digital signatures on exchanged messages.
  • 6. Secure Data Exchange: Once the handshake is complete, the client and server begin exchanging data using the agreed-upon encryption algorithms and keys. This ensures the confidentiality and integrity of the transmitted data.
  • 7. Session Termination: At the end of the session, both parties may agree to terminate the connection, or they can negotiate to reuse the existing session keys for subsequent communications.

using System.Net;
using System.Net.Sockets;
using System.Text;

namespace DeleteLater;

class Program
{
    static void Main(string[] args)
    {
        int serverPort = 54321;

        TcpListener tcpListener = new(IPAddress.Any, serverPort);

        tcpListener.Start();

        Console.WriteLine($"Server listening on port {serverPort} :");

        try
        {
            while (true)
            {
                var tcpClient = tcpListener.AcceptTcpClient();

                Task.Factory.StartNew(() => HandleClient(tcpClient));
            }
        }
        catch (Exception ex)
        {
            Console.WriteLine($"Error: {ex.Message}");
        }
        finally
        {
            tcpListener.Stop();
        }
    }

    static void HandleClient(TcpClient tcpClient)
    {
        try
        {
            Console.WriteLine($"Client connected: {tcpClient.Client.RemoteEndPoint}");

            NetworkStream networkStream = tcpClient.GetStream();

            while (true)
            {
                // Receive messages from the client
                byte[] buffer = new byte[1024];
                int bytesRead = networkStream.Read(buffer, 0, buffer.Length);

                if (bytesRead == 0)
                {
                    // Connection closed by the client
                    break;
                }

                string receivedMessage = Encoding.ASCII.GetString(buffer, 0, bytesRead);
                Console.WriteLine($"Received from {tcpClient.Client.RemoteEndPoint}: {receivedMessage}");
            }
        }
        catch (Exception ex)
        {
            Console.WriteLine($"Error handling client: {ex.Message}");
        }
        finally
        {
            Console.WriteLine($"Client disconnected: {tcpClient.Client.RemoteEndPoint}");

            tcpClient.Close();
        }
    }
}

Common Network Attacks:

Denial-of-Service (DoS) Attacks: These attacks overwhelm a target network or server with a flood of requests, making it unavailable to legitimate users.

Distributed Denial-of-Service (DDoS) Attacks: Similar to DoS attacks, DDoS attacks utilize multiple compromised systems, known as bots, to amplify the attack volume and overwhelm the target even more.

Man-in-the-Middle (MitM) Attacks: These attacks intercept communication between two parties, allowing the attacker to eavesdrop on the traffic and inject malicious data.

Spoofing Attacks: Attackers impersonate legitimate users or systems by altering IP addresses or other identifying information, allowing them to gain unauthorized access or send fraudulent messages.

DNS Hijacking: Attackers modify the DNS records for a domain name, redirecting traffic to malicious websites or servers.

Preventing Network Attacks:

Network Segmentation: Divide the network into smaller, isolated segments to restrict unauthorized access and limit the impact of attacks.

Access Control Lists (ACLs): Implement ACLs to control which devices and users can access specific network resources, based on their identity and privileges.

Intrusion Detection Systems (IDS) and Intrusion Prevention Systems (IPS): Deploy IDS and IPS to monitor network traffic and identify potential attacks, alerting security personnel to take action.

Vulnerability Scanning and Patch Management: Regularly scan networks and systems for vulnerabilities and apply security patches promptly to address known weaknesses.

User Awareness and Training: Educate employees about common attack methods and best practices for safe internet browsing, password management, and data handling.

Data Encryption: Encrypt sensitive data at rest and in transit to protect it from unauthorized access and interception.

Security Awareness for Third-Party Providers: Securely manage access by third-party vendors and service providers to your network, ensuring they follow appropriate security protocols.

DDoS (Distributed Denial-of-Service) attacks typically involve multiple sources, often compromised devices or bots, flooding the target with HTTP requests, malicious traffic, or exploiting vulnerabilities. Motives vary, ranging from disrupting businesses to gaining publicity or extorting money.

To combat DDoS attacks, organizations implement layered defenses: DDoS protection services, firewall configurations, load balancing, and incident response plans.

A rainbow table attack is a technique used to crack password hashes using precomputed hash chains stored in large datasets called rainbow tables.

Rainbow tables consist of many hash chains that essentially help reverse cryptographic hash functions.

They allow crackers to quickly determine the original plaintext passwords from collected password hashes.

Rainbow tables are created by generating common dictionary passwords, hashing them in all combinations, and storing the chained hashes.

During an attack, the captured password hash is compared to the rainbow table to search for a match of the final hash in the chains.

Once a match is found in the rainbow table, the attacker can quickly work backwards to reveal the original plaintext password.

Hash salting refers to the practice of adding a random string of characters, called a salt, to a password before it is hashed. Salting protects against rainbow table attacks.

Symmetric encryption: AES, Triple DES, Blowfish.

MD5 is a one-way function, meaning that it cannot be decrypted back.

It generates the cryptographic hash from data input of arbitrary length like text, binary files etc.

It is extensively used to verify data integrity as minor changes to file generates completely different MD5 hash signatures.

MD5, though popular, is considered cryptographically broken and unsuitable for security applications like SSL certificates due to published exploits.

It is prone to collision attacks where two inputs produce the same hash output. This allows spoofing and impersonation.

Generating MD5 checksums for files like software installers and disk images to verify download integrity by comparing checksums.

MD5 hashes used to identify duplicate data blocks in storage systems like cloud storage for efficient storage and transfers.

Applying MD5 to parts of emails, attachments and contents to flag potential spam through hash comparisons.

REST does not dictate specific methods, but it leverages HTTP methods to perform various operations on resources. The main HTTP methods used in RESTful APIs:

  • GET
    • 1. Retrieves a representation of a resource without modifying it.
    • 2. Idempotent: Yes, multiple identical requests will produce the same result.
    • 3. Safe: Yes, it does not modify the resource.
  • POST
    • 1. Creates a new resource.
    • 2. Idempotent: No, multiple identical requests will create multiple resources.
    • 3. Safe: No, it may have side effects.
  • PUT
    • 1. Updates an existing resource or creates a new resource if it doesn't exist.
    • 2. Idempotent: Yes, multiple identical requests will result in the same state.
    • 3. Safe: No, it modifies the resource.
  • DELETE
    • 1. Removes a resource.
    • 2. Idempotent: Yes, multiple identical requests will result in the same state (resource removal).
    • 3. Safe: No, it modifies the resource.
  • PATCH
    • 1. Partially updates a resource with new data.
    • 2. Idempotent: No, multiple identical requests may result in different states if the patching is based on the current state of the resource.
    • 3. Safe: No, it modifies the resource.
  • OPTIONS
    • 1. Returns the HTTP methods supported by the server for a specified URL.
    • 2. Idempotent: Yes, multiple identical requests will produce the same result.
    • 3. Safe: Yes, it does not modify the resource.
  • HEAD
    • 1. Retrieves the headers that would be returned if the HEAD request's URL was instead requested with the GET method.
    • 2. Idempotent: Yes, multiple identical requests will produce the same result.
    • 3. Safe: Yes, it does not modify the resource.

WSDL stands for Web Services Description Language. It's an XML-based language used to describe the functionalities offered by a web service. In short, WSDL provides a standardized way for clients to understand how to interact with a web service, including the operations it supports, the data types it uses, and the communication protocols it employs. It acts as a contract between the service provider and the consumer, outlining the structure and requirements of the web service.

HTTP (Hypertext Transfer Protocol) requests and responses follow a specific structure that facilitates communication between clients (such as web browsers) and servers.

HTTP Request Structure:

1. Request Line: This line specifies the HTTP method, the requested resource (URI), and the HTTP version. 

GET /index.html HTTP/1.1

2. Headers: Headers provide additional information about the request, such as the user agent, content type, and more. 

Host: www.example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/88.0.4324.150
Content-Type: application/json

3. Body (Optional): Some requests include a body, which contains additional data, such as form input or JSON data. The content type header (Content-Type) in the HTTP request headers specifies the type of content being sent in the body. This JSON data could be part of an HTTP request body when a client is sending user credentials to a server for authentication. 

{
  "username": "example_user",
  "password": "example_password"
}

This is an example of URL-encoded form data. It could be sent in the body of an HTTP request when submitting a form on a web page. 

username=example_user&password=example_password

XML data like this could be included in the body of an HTTP request when exchanging structured data between client and server. 

  <user>
    <username>example_user</username>
    <password>example_password</password>
  </user>

Binary data can be anything from images, files, to any other kind of binary payload. In this case, the body would contain raw binary data.

HTTP Response Structure:

1. Status Line: The status line contains the HTTP version, status code, and a textual description of the status code. 

HTTP/1.1 200 OK

2. Headers: Similar to requests, responses include headers providing additional information, such as content type, content length, etc. 

  Content-Type: text/html
  Content-Length: 1024

3. Body: The body contains the actual content being sent back from the server to the client, such as HTML, JSON, or binary data. 

  HTTP/1.1 200 OK
  Content-Type: text/html

  <!DOCTYPE html>
  <html>
    <head>
      <title>Welcome to Example.com</title>
    </head>
    <body>
      <h1>Hello, World!</h1>
    </body>
  </html>

HTTP Response Structure:

1. Use HTTPS. Encrypt data transmitted between the client and server.

2. Use Token-based Authentication. Clients receive tokens (e.g., JSON Web Tokens - JWT) after successful authentication. These tokens are included in subsequent requests for authentication.

3. OAuth 2.0 is a widely used authorization framework that allows third-party applications to obtain limited access to an HTTP service on behalf of the resource owner.

CORS stands for Cross-Origin Resource Sharing. It's a security feature implemented by web browsers that allows servers to specify which origins (domains) are permitted to access resources from the server via HTTP headers.

Without CORS:
Suppose the API server doesn't implement CORS properly. When the web application on https://example.com tries to make a request to https://api.example.com, the browser blocks the request due to the same-origin policy, which prohibits requests from one origin to another for security reasons. In this case, the browser will throw a CORS error, and the request will fail.

With CORS:
The web application hosted on https://example.com makes a request to https://api.example.com.
The API server responds with the following CORS headers:

  • Access-Control-Allow-Origin: https://example.com
  • Access-Control-Allow-Methods: GET, POST, PUT, DELETE
  • Access-Control-Allow-Headers: Content-Type, Authorization
  • Access-Control-Allow-Credentials: true (if applicable)

The browser checks the CORS headers in the response from https://api.example.com.
Since the Access-Control-Allow-Origin header allows https://example.com, the browser allows the request to proceed.
The browser checks other CORS headers for additional constraints and permissions.

1. The request reaches the load balancer first, which acts as a reverse proxy. The client believes it's communicating directly with the server, but it's actually communicating with the load balancer.

2. The load balancer determines which backend server should handle the request based on predefined algorithms or rules.

3. Once the decision is made, the load balancer forwards the request to the chosen backend server.

4. The backend server processes the request and sends the response back to the load balancer.

5. Finally, the load balancer forwards the response from the backend server back to the client.

6. Load balancers use various algorithms (e.g., Round Robin, Least Connections) to determine how to distribute traffic among backend servers.

Round Robin :
Each new request is sent to the next server in the list, looping back to the first server once all servers have received a request.
Simple to implement, ensures even distribution of traffic among servers, suitable for environments with similar server capacities.
Doesn't consider server load or performance, may still send requests to overloaded servers.

Least Connections :
Directs new requests to the server with the fewest active connections.
Keeps track of the number of active connections on each server and sends new requests to the server with the lowest connection count.
Optimizes server utilization by distributing load based on current connections, ensures efficient resource allocation.
Requires monitoring of active connections on each server, may not account for server capacity or performance differences.

7. They can perform health checks on backend servers to ensure they are available and responsive before forwarding requests.

8. Some advanced load balancers offer features like session persistence, where subsequent requests from the same client are sent to the same backend server.

Authentication: answers the question "Who are you?"

Example: When you log in to your email account, you provide your username and password. The email service authenticates your credentials to verify that you are the legitimate owner of the account.

Authentication is the process of verifying the identity of a user, device, or system.

Authorization: answers the question "What are you allowed to do?"

Example: After successfully authenticating yourself on a company's internal network, the authorization process determines which files, applications, or services you are allowed to access based on your job role or security clearance level. An HR manager may have access to employee records, while a developer may only have access to certain code repositories.

Authorization is the process of determining what actions or resources an authenticated entity is permitted to access or perform. It defines the level of access and privileges granted to a user, device, or system based on their role, responsibilities, or security clearance.

SaaS (Software as a Service): SaaS provides software applications over the internet, hosted and maintained by the service provider. Users access the software through web browsers or lightweight client applications.

Example: Google Workspace (formerly G Suite), Microsoft Office 365, Salesforce, Dropbox, and Zoom are examples of SaaS offerings.

PaaS (Platform as a Service): PaaS provides a complete platform for developing, testing, deploying, and managing applications. It includes the necessary infrastructure, such as servers, databases, and development tools, managed by the service provider.

Example: AWS Elastic Beanstalk, Google App Engine, Heroku, and Microsoft Azure App Service are examples of PaaS platforms.

IaaS (Infrastructure as a Service): IaaS provides virtualized computing resources, such as servers, storage, networks, and other infrastructure components, over the internet. Users can provision and manage these resources as needed.

Example: Amazon Web Services (AWS) EC2 (Elastic Compute Cloud), Microsoft Azure Virtual Machines, Google Compute Engine, and DigitalOcean Droplets are examples of IaaS offerings.

SaaS: Users access and use software applications over the internet.

PaaS: Developers build, run, and manage applications on a platform provided by the service provider.

IaaS: Organizations rent computing resources (servers, storage, networks) on a pay-as-you-go basis from service providers.
